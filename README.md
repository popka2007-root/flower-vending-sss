# Платформа управления цветочным вендинговым аппаратом (Flower Vending System)

**Flower Vending System v0.2.0** — это передовая платформа управления киоском по продаже премиальных букетов цветов. Система разработана в режиме "первичности симулятора" (Simulator-first), что позволяет проводить глубокое тестирование логики без необходимости реального оборудования, но при этом она полностью готова к работе с реальным "железом" в режиме киоска (kiosk mode) через сенсорный экран.

Система взаимодействует с купюроприёмником, моторами выдачи товара, обеспечивает контроль температуры и включает продвинутые механизмы обработки сбоев.

---

## 1. Ключевые возможности

*   **Интерактивный каталог товаров:** Удобный интерфейс выбора букетов, корзина и оформление заказа через сенсорный экран.
*   **Приём наличных:** Интеграция с купюроприёмником (JCM DBV-300-SD) с проверкой возможности выдачи сдачи.
*   **Выдача товара:** Контроль моторных слотов и дверец выдачи товара через ESP32 Arduino.
*   **Климат-контроль:** Мониторинг и контроль температуры внутри витрины для поддержания свежести цветов.
*   **Отказоустойчивость (Crash recovery):** Журналирование намерений и результатов (intent/outcome) для каждого аппаратного действия. При перезапуске система восстанавливает прерванные транзакции.
*   **Сервисный режим:** Защищенный PIN-кодом доступ для операторов (дашборд, диагностика, управление инвентарем).
*   **Интеграции:** Клиент для обмена данными с 1С и публикация телеметрии.

## 2. Стек технологий

Проект построен с использованием современных и надежных технологий, придерживаясь строгих архитектурных стандартов:

*   **Язык:** Python 3.11+
*   **Пользовательский интерфейс (UI):** PySide6 (Qt6)
*   **Асинхронность:** `asyncio` для эффективной работы с устройствами и сетью.
*   **Конфигурация:** Pydantic v2 + PyYAML (надежная типизация настроек).
*   **База данных:** SQLite (с режимами WAL + FULL sync для предотвращения потери данных).
*   **Взаимодействие с оборудованием:** `pyserial` (UART), интеграция с ESP32 Arduino (C++) для управления моторами/сенсорами.
*   **Сборка и упаковка:** Hatchling, PyInstaller, Inno Setup, AppImage.
*   **Качество кода:** Ruff (линтер и форматтер), mypy (строгая типизация), pre-commit.
*   **Тестирование:** pytest, pytest-asyncio (более 100 тестов: unit, integration, recovery, e2e).

## 3. Архитектура проекта (Clean Architecture)

Проект использует строгое разделение по принципам **Clean Architecture** (Чистая Архитектура), где зависимости направлены исключительно внутрь, к доменному ядру.

```text
┌──────────────────────────────────────────────────────┐
│                   UI LAYER (PySide6)                  │
│  KioskWindow → ScreenWidgets → KioskPresenter        │
└──────────────────────┬───────────────────────────────┘
                       │ async вызовы через asyncio
┌──────────────────────┴───────────────────────────────┐
│              APPLICATION FACADE                       │
│  UiApplicationFacade (мост между UI и Core)           │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────┴───────────────────────────────┐
│              APPLICATION CORE                         │
│  ApplicationCore → CommandBus + EventBus + FSM        │
│  Orchestrators: VendingController, PaymentCoordinator,│
│  TransactionCoordinator, RecoveryManager и др.        │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────┴───────────────────────────────┐
│              DOMAIN CORE (zero external imports!)     │
│  Entities, Value Objects, Aggregates, Domain Events,  │
│  Domain Commands, Exceptions                          │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────┴───────────────────────────────┐
│              INFRASTRUCTURE                           │
│  SQLite, Repositories, Journal, Config (Pydantic),    │
│  Logging (JSONL)                                      │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────┴───────────────────────────────┐
│              DEVICES & INTEGRATION                    │
│  ABC interfaces → DBV300SD / Arduino / Simulators    │
│  1C Client, Telemetry Publisher                      │
└──────────────────────────────────────────────────────┘
```

### Правила изоляции слоев
- **Domain Core:** Сердце системы. Здесь нет импортов фреймворков или библиотек (кроме стандартных). Описывает бизнес-правила.
- **Application Core:** Конечный автомат (FSM) из 21 состояния, который строго контролирует процесс продажи. Зависит только от Domain.
- **Infrastructure и Devices:** Реализуют интерфейсы, определенные в ядре. Включают симуляторы для разработки без реального "железа".
- **UI Layer:** Зависит только от Application Facade, без прямого доступа к базе данных или логике.

## 4. Конечный автомат (FSM)

Система работает на основе конечного автомата, состоящего из 21 состояния. Переходы строго контролируются и валидируются `StateMachineEngine`.

**Основной жизненный цикл продажи:**
```text
BOOT → IDLE → PRODUCT_SELECTED → CHECKING_AVAILABILITY → CHECKING_CHANGE
→ WAITING_FOR_PAYMENT → ACCEPTING_CASH → PAYMENT_ACCEPTED
→ DISPENSING_CHANGE → DISPENSING_PRODUCT → OPENING_DELIVERY_WINDOW
→ WAITING_FOR_CUSTOMER_PICKUP → CLOSING_DELIVERY_WINDOW → COMPLETED → IDLE
```

**Особые состояния:**
- При любых сбоях оборудования: переход в `FAULT` → `RECOVERY_PENDING` (или `OUT_OF_SERVICE`).
- Сервисный режим: `IDLE` → `SERVICE_MODE` (по вводу PIN-кода).

## 5. Отказоустойчивость и Персистентность (Crash Recovery)

Система устойчива к потере питания благодаря механизму **Intent/Outcome Journal**.

- **Атомарность операций:** Перед каждым аппаратным действием (например, прокруткой мотора выдачи) в SQLite записывается **намерение (intent)**.
- После успешного завершения записывается **результат (outcome)**.
- **RecoveryManager:** При перезагрузке система ищет незавершённые намерения. Если мотор начал выдачу, но отключилось питание, система обнаружит незавершенный `intent` и перейдет в режим `RECOVERY_PENDING`, требуя вмешательства оператора (или автоматически восстановит безопасные операции, такие как отмена).

БД SQLite настроена на `journal_mode=WAL` и `synchronous=FULL` для гарантии сохранности данных. Логирование ведется в формате JSON Lines (`.jsonl`) через отдельный поток (`QueueHandler`), исключая блокировку основного потока или интерфейса.

## 6. Взаимодействие с оборудованием

Связь с физическими устройствами осуществляется через последовательный порт (`pyserial`):

1. **ESP32 Arduino Firmware:** Управляет реле моторов, дверец, датчиками температуры (DS18B20). Взаимодействие по UART (115200 baud). Формат команд текстовый: `VEND_SLOT N`, `STATUS`, `TEMP`.
2. **JCM DBV-300-SD:** Купюроприёмник по протоколу JCM. Использует бинарный фреймовый протокол (9600 baud). Имеет асинхронный цикл опроса (`poll loop`).

В случае потери соединения (например, выдернули USB-кабель), система автоматически переводит устройство в состояние `FAULT` и блокирует продажи. При восстановлении соединения — автоматически переподключается.

## 8. Как добавить новую фичу

Процесс добавления новой бизнес-фичи строго регламентирован архитектурой:

1. Добавьте событие (Domain Event) в `domain/events/`.
2. При необходимости добавьте команду (Domain Command) в `domain/commands/`.
3. Реализуйте логику в оркестраторе в `app/orchestrators/`.
4. Зарегистрируйте команду и обработчик в сборщике `app/bootstrap.py`.
5. Добавьте ViewModel в `ui/viewmodels/` и экран в `ui/views/`.
6. Зарегистрируйте экран в `kiosk_window.py`.
7. Покройте изменения тестами: `unit` → `integration` → `e2e`.

## 10. Технический аудит и Roadmap

По результатам глубокого технического аудита, проведенного с точки зрения энтерпрайз-стандартов, отказоустойчивости и готовности к production (как софт для реального вендингового оборудования), представлен следующий анализ и план действий:

### Краткий обзор архитектуры
*   **Стек технологий:** Python 3.11+, PySide6 для UI, asyncio для конкурентности, Pydantic, SQLite.
*   **Тип архитектуры:** Модульный монолит с выраженной попыткой следовать Clean Architecture (Слои: domain, app, infrastructure, ui, devices).
*   **Соответствие Best Practices:** Структура директорий хорошая. Направление зависимостей (UI -> App -> Domain) в целом соблюдается. Однако, реализация хромает. Архитектура выглядит как "Clean Architecture на бумаге", но на практике изобилует God-объектами, блокирующим I/O в асинхронном контексте и грубым проглатыванием ошибок оборудования.

### Оценка качества: 4.5 из 10
Проект на данный момент находится на стадии "MVP-прототипа".

*   **Чистота (Clean Code) — 4/10:** Нарушены принципы SRP (Single Responsibility) и YAGNI. Присутствуют гигантские классы-полотна, которые сложно читать и поддерживать.
*   **Безопасность (Reliability) — 4/10:** Отказоустойчивость реализована плохо. Найдено более 86 мест с конструкцией `except Exception:`. При ошибке оборудования автомат будет молча вести себя непредсказуемо, скрывая реальные stack-trace ошибки.
*   **Производительность (Performance) — 3/10:** Главная архитектурная ошибка: асинхронный event loop блокируется синхронными вызовами к SQLite (`SQLiteDatabase`). Любая задержка диска I/O вызовет полное зависание сенсорного UI.
*   **Масштабируемость (Scalability) — 6/10:** Добавление новых устройств продумано хорошо (через адаптеры), но расширение бизнес-логики затруднено из-за перегруженности оркестраторов.
*   **Покрытие тестами/поддерживаемость — 5/10:** Имеется множество тестов (~160), но наличие падающих тестов (например, в `TransactionCoordinator`) свидетельствует о необходимости повышения дисциплины разработки.

### Детальный разбор слабых мест (Критический анализ)

🔴 **Блокирующий дисковый I/O (Критический bottleneck)**
Синхронный `sqlite3.connect()` и `threading.RLock()` внутри asyncio приложения (`infrastructure/persistence/sqlite/database.py`). Это вызывает зависания UI при дисковых операциях.

🔴 **Обработка ошибок (Антипаттерн Pokemon Exception Handling)**
Почти 90 блоков `except Exception:` в адаптерах и оркестраторах. Глушение исключений без проброса или логирования (`logger.exception()`) скрывает причины отказов (например, купюроприемника).

🔴 **God Objects и нарушение SOLID**
*   `bootstrap.py` (>1100 строк): выполняет DI, БД, мокинг, логирование.
*   `kiosk_presenter.py` (>870 строк): управляет сессией, корзиной, навигацией.
*   `catalog_screen.py` (>830 строк): смешивает UI, анимации и бизнес-логику.

🔴 **Гонки состояний (Race Conditions) в транзакциях**
В `app/orchestrators/transaction_coordinator.py` падают тесты, связанные с обработкой терминальных статусов (например, автомат блокируется при `FAULTED` транзакции).

### Roadmap: Пошаговый план "Как довести до идеала" (Краткосрочный план на 5 дней)

🔥 **Очередь 1 (Critical) - Устойчивость и баги**
1.  **Починить I/O:** Обернуть текущие синхронные вызовы `SQLiteDatabase` через `ThreadPoolExecutor` (или перейти на `aiosqlite`). Это мгновенно устранит фризы UI.
2.  **Исправить тесты и логику:** Устранить race conditions в `TransactionCoordinator`. Разрешить очистку `_active_transaction_id`, если автомат может продолжить работу после `FAULTED`.
3.  **Уничтожить `except Exception:`:** Заменить слепое глушение на отлов конкретных исключений оборудования (например, `serial.SerialException`) с обязательным `logger.exception()`.

🟡 **Очередь 2 (Medium) - Архитектурный долг**
1.  **Рефакторинг `bootstrap.py`:** Разбить на `di/infrastructure.py`, `di/domain.py`, `di/ui.py`.
2.  **Разгрузка `kiosk_presenter.py`:** Выделить паттерн Router для навигации и SessionManager для управления корзиной.

🟢 **Очередь 3 (Low) - Косметика и CI**
1.  **Жесткие правила линтера:** Включить правило Ruff `BLE001` (Do not catch blind exception).
2.  **Вынос стилей UI:** Перенести захардкоженные стили (цвета, градиенты) в QSS-файлы.
